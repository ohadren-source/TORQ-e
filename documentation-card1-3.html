1. System Overview

TORQ-e is the New York State Medicaid Clarity System. Five cards, five audiences, one unified identity flowing through all of them. Cards 1, 2, and 3 serve external users (members, providers, plan administrators); Cards 4 and 5 serve government oversight and forensic investigation.

This document covers Cards 1–3. For Cards 4 and 5, see documentation-card4-5.html.

The unified identity model

  • UMID — Universal Member ID. Follows a member across every Medicaid system in New York and federally. Never changes.
  • UPID — Universal Provider ID. Follows a provider across every MCO panel and FFS enrollment.
  • UHWP — Universal Health & Wellness Program tracking. Plan-level identity for network and enrollment administration.

Each card accepts the user's identity, queries real public repositories via the background crawler, and returns answers with confidence scores and live source URLs.

2. Card 1 — UMID (Member Portal) C1

Card 1 is the public-facing chat for Medicaid members. Members can ask plain-language questions about their eligibility, recertification timing, benefit details, and document requirements.

Available tools

  • lookup_member — Look up member by UMID, name+DOB, or SSN. Returns identity record with confidence score.
  • check_eligibility — Check if member's coverage is active, expiration date, plan affiliation.
  • check_recertification — When is the next renewal? What documents are needed?

What members see

The chat opens with a welcome message listing common requests. Once a member is identified (UMID in session), the chat personalizes the welcome and uses the UMID for every lookup automatically — no asking for ID repeatedly.

If no UMID is in the session, the chat falls back to general guidance and prompts the user to log in for personalized data. This is intentional — the system never fabricates account-specific answers.

River Path on Card 1

  1. Primary: NY State Medicaid database (MPI) — confidence 0.98, real-time
  2. Secondary: SSA wage records — confidence 0.85, federal data
  3. Tertiary: Household enrollment records — confidence 0.70, manual queue
  4. Escalation: All three failed → return graceful failure with reason and next step.

3. Card 2 — UPID (Provider Portal) C2

Card 2 is the provider-facing chat. Providers can look up enrollment status across FFS and every MCO, validate claims before submission, and route claims to the correct payer.

Available tools

  • lookup_provider — Find provider by NPI, name, or specialty. Returns UPID and identity record.
  • check_enrollment — Returns FFS status, every MCO panel affiliation, license validity, and a confidence score.
  • validate_claim — Checks a claim for the most common errors (NPI/Tax ID mismatch, missing prior auth, expired credentialing, invalid procedure code) before submission.

How claim routing works

Providers often submit Medicaid managed-care claims to eMedNY by mistake (or vice versa) and wait 60 days wondering why payment didn't come. Card 2 looks up the member's plan, then tells the provider which portal to submit through and what tax ID to use. This is the highest-leverage feature for reducing provider abandonment of Medicaid.

4. Card 3 — UHWP (Plan/Network Admin Dashboard) C3

Card 3 is the plan administrator chat. Plan admins monitor network-wide metrics across six dimensions, drill into specific MCOs or regions, and compare plans side-by-side.

The six dimensions

DimensionWhat it measures
Network AdequacyProvider counts, access gaps, geographic coverage
Plan EnrollmentMember trends, churn, population shifts
Formulary CoverageDrug tier placement, exceptions, denials
Claims AcceptanceDenial rates, processing times, outliers
Network ChangesProvider additions/terminations, disruptions
MCO AvailabilityPlan options by region, competitive landscape

Card 3 Always-External Rule

Plan administrative data is always external to NY state systems. It lives in MCO systems, plan databases, and federal registries. Every Card 3 response renders traffic light + live URL for every external claim. Format: 🟢 HIGH | [Source Name] | https://.... Never a bare URL. Never a confidence light without a URL.

Available tool

  • query_plan_metrics — Returns all six dimensions with values, confidence scores, and source citations. Optional metric_type filter to focus on one dimension; optional state (defaults to NY).

5. Confidence Scoring & Source Vetting

Every claim returned by any TORQ-e card carries a confidence score. The score is domain-based with quality bonuses, capped at 0.95 to reflect that no public-data signal is ever 100% authoritative.

DomainBase scoreReasoning
emedny.org0.85Official, daily updates
omig.ny.gov0.80Audit / compliance / PERM
health.data.ny.gov0.75Open-data portal
its.ny.gov0.70System / infrastructure
other0.60Unknown source

Bonuses: +0.10 if the source is a structured table with extracted text; +0.05 if a plain page with extracted text. Cap: 0.95.

Clarity light thresholds

  • 🟢 Green (≥65%) — Coherent. Proceed.
  • 🟡 Yellow (50–64%) — Monitoring. Review before reporting.
  • 🔴 Red (<50%) — Incoherent. Signal too fragmented to reason from.

6. Chat Interface & Streaming Behavior

All three cards use the same SSE streaming pipeline:

  1. User types a message and clicks send (or presses Enter).
  2. Frontend POSTs to /api/chat/stream with {message, userType, cardNumber, sessionId}.
  3. Backend streams Claude's response as data: {"text": "..."} SSE events.
  4. Frontend buffers chunks line-by-line, displays the placeholder ("Considering Available Data...") until the first real token arrives, then replaces it with streaming text.
  5. At end of stream, the full text is rendered through markdown-it and sanitized via DOMPurify.

Tool calls

Each card exposes its own tool surface. When Claude decides a question requires data, it calls the appropriate tool (e.g., query_plan_metrics for Card 3) and waits for the tool result before continuing the response. The user sees this as a brief pause in streaming followed by data-grounded output with source URLs.

7. Privacy & HIPAA Boundaries

  • All requests use TLS 1.3.
  • Member PII (SSN, DOB, full address) is never logged in plain text. Audit logs record that a query happened, not the contents.
  • Cards 1 and 2 enforce minimum-necessary — they fetch only the fields needed to answer the user's question.
  • Card 3 is aggregate-only at the plan level; no individual member records.
  • Cards 1, 2, 3 do not have full-fidelity PII access. Forensic investigation requiring SSNs and member identities escalates to Card 5 (UBADA) under separate access controls.

8. Troubleshooting

The chat button does nothing when I click

Most common cause: browser cache serving an older HTML. Hard refresh (Ctrl+Shift+R or Cmd+Shift+R). If still unresponsive, try in a private/incognito window. The button is initially disabled until you type a character — type once and the button enables.

"Considering Available Data..." hangs without a response

Background crawler may still be initializing (~30s after deploy). Wait a moment. If it persists more than a minute, the Anthropic API may be having a hiccup. Try the canary endpoint: open the browser console on any chat-card page and run helloClaude("test"). If that returns a response, the chat router has a separate issue; if not, the API itself is unreachable.

Response shows raw markdown (** , | --- |)

Means markdown-it didn't load. Hard refresh to retry the CDN. If it persists in incognito, contact support.

"Welcome back, UMID" or "I recognize your UMID: UMID"

Session-data artifact from a test login. Clear browser session storage for this domain or use incognito for a clean session.

Claim was validated but rejected by the payer

Card 2's validate_claim catches the most common errors but cannot verify payer-specific edits or member-level eligibility at the date of service. Run the validation, fix what's flagged, then submit. If still rejected, the rejection reason from the payer points to the specific edit.
