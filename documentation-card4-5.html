🏗️ System Architecture

TORQ-e Overview

TORQ-e (TORQ-Enhanced) is a HIPAA-compliant Medicaid Clarity System built on a five-card architecture. Each card serves a distinct stakeholder group with specific data access patterns and operational workflows.

Five-Card Architecture

Card Code Stakeholder Data Access Status
Card 1 UMID Member Eligibility Self (own records only) ✅ LIVE
Card 2 UPID Provider System Enrolled members' claims ✅ LIVE
Card 3 UHWP Plan Administrator Administrative oversight ⏭️ Planned
Card 4 USHI Government Stakeholder Aggregate metrics only 🔨 In Development
Card 5 UBADA Data Analyst / Fraud Investigation Complete dataset (with audit) ✅ Backend Ready

Technology Stack

  • Backend: FastAPI (Python) with asynchronous request handling
  • Database: SQLAlchemy ORM with audit trail persistence
  • Security: Fernet encryption (AES-128), role-based access control, audit logging
  • Frontend: Responsive HTML5 with vanilla JavaScript
  • AI Integration: Claude API for intelligent analysis and investigation
  • Compliance: HIPAA-compliant logging, minimum-necessary data access, de-identification

Core Principles

  • Least Privilege: Users access only the minimum data necessary for their role
  • Immutable Audit Trail: All actions are logged with WHO, WHAT, WHEN, WHY for regulatory compliance
  • Data Governance: Centralized source management with version control and quality scoring
  • Confidence Scoring: Quality × Completeness × Freshness (0.0-1.0) for data reliability assessment
  • River Path Algorithm: Multi-source lookup cascade with graceful degradation for fault tolerance

🏛️ Card 4 (USHI) - Government Stakeholder

Business Purpose

Card 4 serves government stakeholders (state health officials, regulatory agencies, public health administrators) who need aggregate-level insights for policy-making, compliance monitoring, and program oversight. All data access is aggregate-only with no individual member or provider PII visible.

Key Capabilities

  • Aggregate Metrics: Enrollment trends, utilization rates, cost patterns, network health
  • Quality Monitoring: Provider performance, member satisfaction, compliance metrics
  • Governance Interface: View and flag issues in source data, generate investigation recommendations
  • Audit Trail Access: Transparent logging of all Card 4 actions for accountability
  • Escalation Workflow: Flag data quality issues for Card 5 investigation

Data Access Pattern

Key Constraint: Card 4 is aggregate-only. No individual member names, SSNs, provider NPIs, or claim details are visible. All data is aggregated, de-identified, and summarized by category (region, service type, date range, etc.).

Viewable Metrics Categories

  • Enrollment: Total members, monthly trends, demographic breakdowns
  • Utilization: Service usage rates, provider network size, claim volume
  • Financial: Aggregate spend, cost-per-member trends, budget forecasts
  • Quality: Provider performance scores, member satisfaction indices, compliance rates
  • Governance: Data quality scores, source reliability, flagged issues

Role-Based Permissions

Government Stakeholder users have the following permissions:

  • ✅ VIEW_AGGREGATE (all metrics categories)
  • ✅ FLAG_ISSUE (mark data quality concerns)
  • ✅ STRIKE_SOURCE (recommend source removal/replacement)
  • ❌ VIEW_INDIVIDUAL (no individual records)
  • ❌ VIEW_PII (no names, SSNs, NPIs)
  • ❌ CREATE_INVESTIGATION (escalate to Card 5 instead)

Example Use Cases

  • Monthly oversight: Review enrollment trends, identify growth patterns or sudden changes
  • Quality assurance: Monitor provider performance metrics, flag underperforming networks
  • Budget forecasting: Analyze aggregate spend trends, predict future cost trajectories
  • Compliance monitoring: Check data quality scores, ensure all sources meet reliability thresholds
  • Issue escalation: Flag suspicious patterns (outliers, data gaps) for Card 5 investigation

🔬 Card 5 (UBADA) - Data Analyst & Fraud Investigation

Business Purpose

Card 5 serves authorized data analysts and fraud investigators who require complete dataset access to detect anomalies, investigate suspicious patterns, and recommend data corrections. Full PII is accessible under strict audit logging and role-based permissions.

Key Capabilities

  • Claims Exploration: Deep-dive analysis of individual claims with full member/provider context
  • Outlier Detection: Statistical analysis (Z-scores) to identify unusual patterns
  • Relationship Navigation: Graph-based exploration of member-provider-claim networks
  • Investigation Creation: Launch formal investigations with documentation and evidence trails
  • Data Correction: Request amendments to erroneous records with justification
  • Immutable Audit Trail: Complete WHO/WHAT/WHEN/WHY logging with meta-audit (accessing audit trail is itself logged)

Data Access Pattern

Complete Data Access: Card 5 users see full individual records including member names, SSNs, provider NPIs, claim amounts, and service details. Access is governed by role-based permissions and subject to audit logging.

Confidence Scoring System

All Card 5 outputs include confidence scores (0.0 to 1.0) calculated as:

Confidence = Quality × Completeness × Freshness
  • Quality (0.0-1.0): Source reliability, data validation status
  • Completeness (0.0-1.0): Proportion of required fields populated
  • Freshness (0.0-1.0): Time since data last verified (1.0 = within 7 days, degrading to 0.0 at 180+ days)

Color-Mapped Confidence

  • 🟢 High (0.70-1.0): Reliable data, proceed with confidence
  • 🟡 Medium (0.40-0.69): Partial reliability, use with caution
  • 🔴 Low (<0.40): Questionable reliability, verify independently

Role-Based Permissions

Data Analyst / Fraud Investigator users have full permissions:

  • ✅ VIEW_INDIVIDUAL (all individual records)
  • ✅ VIEW_PII (names, SSNs, NPIs, claim details)
  • ✅ CREATE_INVESTIGATION (formal investigation workspace)
  • ✅ REQUEST_CORRECTION (propose data amendments)
  • ✅ APPROVE_CORRECTION (review and approve corrections)
  • ✅ FLAG_ISSUE (mark data quality concerns)

Example Use Cases

  • Fraud Detection: Identify patterns of unusual billing (member ID misuse, provider overbilling)
  • Data Quality Investigation: Research incomplete or inconsistent records, recommend corrections
  • Anomaly Analysis: Use outlier detection to find members with unusual claim frequencies or amounts
  • Network Analysis: Explore relationships between members, providers, and services to identify suspicious patterns
  • Investigation Case Work: Create formal investigations with documented evidence trails and audit logging

🔒 HIPAA Compliance & Security

Regulatory Framework

TORQ-e implements HIPAA Security Rule requirements including administrative safeguards, physical safeguards, technical safeguards, and organizational policies.

Minimum Necessary Principle

Core HIPAA Requirement: The system enforces that each role accesses only the minimum amount of data necessary to perform their function. Card 4 receives aggregate data only. Card 5 receives complete data but only for authorized investigators.

Data Access Enforcement

  • Card 1 & 2 (Members/Providers): Only their own records visible
  • Card 4 (Government): Aggregate metrics only, no individual PII
  • Card 5 (Analysts): Complete records with strict audit logging

De-Identification & Masking

The system supports both data masking and encryption for sensitive fields:

  • Masking (viewable but obscured): SSN → XXX-XX-6789, NPI → *****7890, Name → J*** D***
  • Encryption (not viewable): AES-128 Fernet encryption for complete fields
  • Rule-Based: Masks are applied according to role permissions and business context

Immutable Audit Trail

Every action is logged with immutable records containing:

Field Description Example
WHO User ID and role analyst_jane, DataAnalyst
WHAT Action type and affected entities QUERY_CLAIMS, Member ID 12345
WHEN Timestamp with timezone 2026-04-25T14:32:15Z
WHY Justification for the action "Fraud investigation case INV-2026-001"

Meta-Audit Concept

Accessing the audit trail itself is logged as an action. This prevents unauthorized review of audit logs and creates a complete chain of custody for all PII access.

Example: If analyst_jane views the audit log for a sensitive claim, that action (VIEW_AUDIT_TRAIL on claim_12345) is itself logged with analyst_jane as the actor.

Encryption & Key Management

  • Algorithm: Fernet (AES-128) symmetric encryption
  • Key Storage: TORQ_E_ENCRYPTION_KEY environment variable (rotate periodically)
  • Encrypted Fields: SSN, member names, provider NPIs in sensitive contexts
  • Key Rotation: Support for re-keying data when keys are compromised

Role-Based Access Control (RBAC)

The system enforces RBAC through a permission matrix:

Roles & Permissions Matrix

Role VIEW_AGGREGATE VIEW_INDIVIDUAL VIEW_PII CREATE_INVESTIGATION REQUEST_CORRECTION
Member Own Only Own Only Own Only ❌ Own Records
Provider Own Network Own Network Own Network ❌ Own Claims
Government Stakeholder ✅ All ❌ ❌ ❌ ❌
Data Analyst / Investigator ✅ All ✅ All ✅ All ✅ ✅

Data Retention & Purging

  • Active Records: Retained for operational period (typically 3+ years)
  • Audit Trail: Retained for 7 years (HIPAA requirement)
  • Purging: Compliance automated process with documented retention schedules
  • Right to Erasure: Members can request deletion of their personal data (with regulatory exceptions)

Security Monitoring & Incident Response

  • Anomaly Detection: Alerts for unusual access patterns (bulk downloads, after-hours access, etc.)
  • Incident Logging: All security events recorded with timestamp and context
  • Breach Notification: Automatic notification workflow if unauthorized access detected
  • Regular Audits: Quarterly review of access logs and permissions

🔄 Data Governance & Quality Management

Source Management System

TORQ-e implements centralized source management for all data inputs, with version control, quality tracking, and reliability scoring.

Data Source Lifecycle

  • Registration: New sources are registered with metadata (name, owner, contact, frequency)
  • Validation: Initial data quality assessment (schema compliance, completeness, accuracy)
  • Monitoring: Continuous quality scoring during operational use
  • Flagging: Automatic alerts when quality degrades below thresholds
  • Remediation: Governance team works with source owners to resolve issues
  • Deprecation: Graceful retirement of unreliable sources with data migration plans

Confidence Scoring Framework

Each piece of data receives a confidence score combining three dimensions:

Confidence Score = Quality × Completeness × Freshness

Result: 0.0 (unreliable) to 1.0 (highly reliable)

Quality Dimension

  • Source reliability track record
  • Schema compliance percentage
  • Historical error rate
  • Validation rule pass rate

Completeness Dimension

  • Required fields populated
  • No null values in critical columns
  • Data type consistency
  • Reference integrity (foreign keys valid)

Freshness Dimension

  • Days since last verification: 1.0 (0-7 days) → 0.5 (30 days) → 0.0 (180+ days)
  • Publication lag from source
  • Staleness of referenced data

River Path Algorithm

Multi-source lookup cascade for fault tolerance and data coverage:

For member_lookup(member_id): 1. Try Primary Source (highest quality) 2. If not found, cascade to Secondary Source A 3. If not found, cascade to Secondary Source B 4. If not found, try historical archive 5. If still not found, return "not found" with audit note 6. For each step, record confidence score 7. Return best-confidence result with source lineage
Advantage: Continues operation even if primary source fails or lags. Users always see best-available data with confidence context.

Governance Log

Centralized log of all governance actions including flagging, striking sources, and remediation:

  • Flagged Issues: Card 4 users flag data quality concerns
  • Source Strikes: Formal removal of unreliable sources
  • Corrections: Card 5 data corrections with approval status
  • Remediation Tracking: Follow-up actions and resolution status
  • Audit Trail: Complete WHO/WHAT/WHEN/WHY for all governance actions

Data Quality Thresholds

Threshold Action Description
Confidence > 0.70 ✅ Approve High quality, use with confidence
Confidence 0.40-0.70 ⚠️ Caution Medium quality, verify independently
Confidence < 0.40 🚫 Flag Low quality, escalate to Card 5
Completeness < 70% 🚫 Alert Source failing validation rules

⚙️ Operations & Deployment

System Requirements

  • Python: 3.10 or higher
  • Database: PostgreSQL 13+ (or SQLAlchemy-compatible database)
  • FastAPI: 0.104+
  • Python Packages: fastapi, sqlalchemy, cryptography, pydantic
  • Environment: Linux/macOS/Windows with Docker support recommended

Installation & Setup

# Clone repository git clone https://github.com/your-org/torqe.git cd torqe # Install dependencies pip install -r requirements.txt # Set up environment variables export TORQ_E_DATABASE_URL="postgresql://user:password@localhost/torqe" export TORQ_E_ENCRYPTION_KEY="your-encryption-key" export TORQ_E_CLAUDE_API_KEY="your-claude-api-key" # Initialize database python -m database init_db # Start server uvicorn main:app --host 0.0.0.0 --port 8000 --reload

Docker Deployment

# Build image docker build -t torqe:latest . # Run container docker run -e TORQ_E_DATABASE_URL="postgresql://..." \ -e TORQ_E_ENCRYPTION_KEY="..." \ -e TORQ_E_CLAUDE_API_KEY="..." \ -p 8000:8000 \ torqe:latest

Environment Configuration

Variable Description Required
TORQ_E_DATABASE_URL Database connection string (PostgreSQL recommended) ✅ Yes
TORQ_E_ENCRYPTION_KEY Fernet encryption key for PII fields ✅ Yes
TORQ_E_CLAUDE_API_KEY Anthropic Claude API key for intelligent analysis ✅ Yes
TORQ_E_API_VERSION API version (default: 1.0.0) ❌ No
TORQ_E_LOG_LEVEL Logging level: DEBUG, INFO, WARNING, ERROR (default: INFO) ❌ No

Monitoring & Logging

  • Application Logs: FastAPI logs all requests and responses with timestamps
  • Audit Logs: SQLAlchemy ORM captures all data access in audit_trail_entry table
  • Error Tracking: Stack traces logged with context for debugging
  • Health Endpoint: GET /api/card5/health returns system status
  • Performance Metrics: Query execution times logged for optimization
# Check system health curl http://localhost:8000/api/card5/health Response: { "status": "operational", "database": "connected", "services": ["card1", "card2", "card4", "card5"], "timestamp": "2026-04-25T14:40:00Z" }

Backup & Disaster Recovery

  • Daily Backups: Automated database backups with retention (30 days minimum)
  • Encryption Key Backups: Secure backup of TORQ_E_ENCRYPTION_KEY in vault
  • Recovery Testing: Monthly restore tests to verify backup integrity
  • RTO/RPO: Recovery Time Objective 4 hours, Recovery Point Objective 24 hours

Performance Optimization

  • Database Indexing: Indexes on member_id, claim_id, provider_npi, created_at
  • Query Caching: Cache aggregate metrics (1-hour TTL)
  • Pagination: Large result sets paginated (default 100 records per page)
  • Connection Pooling: SQLAlchemy connection pool with 10 connections

Common Issues & Resolution

Permission Denied Error

Problem: "User lacks permission for this action"
Solution: Verify your role and the specific permission required. Card 4 users cannot create investigations (Card 5 only). Check with your administrator if permissions should be adjusted.

Audit Trail Not Found

Problem: Cannot retrieve audit log for an action
Solution: Audit trails are immutable and retained for 7 years. Older records may be archived. Check the action timestamp and contact support if record should exist.

Data Discrepancies

Problem: Data shows different values in different cards
Solution: Check confidence scores for each source. Different cards may show different subsets of data due to access restrictions. See source documentation for freshness.

Investigation Creation Failed

Problem: Cannot create investigation
Solution: Only Card 5 (Data Analyst) role can create investigations. Card 4 users should escalate via the Flag Issue workflow.
