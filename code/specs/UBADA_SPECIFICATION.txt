1. DEFINITION
================================================================================

UBADA (Universal Business Analyst and Data Analyst) is the single, persistent,
unique identifier that represents an authorized analyst accessing TORQ-e
systems for data analysis, fraud detection, reporting, and insight generation.

UBADA represents the analytical role, skill level, data access scope, and
analysis objectives for users who:
- Detect fraud and anomalies in claims
- Generate program efficiency reports
- Analyze provider/member patterns
- Identify waste and inefficiency
- Track quality metrics and outcomes
- Support program evaluation and improvement
- Conduct ad-hoc analyses

UBADA is generated ONCE when an analyst role is authorized and persists for
the duration of their authorized analytical access.

Note: UBADA combines Business Analysts (program-focused) and Data Analysts
(technical-focused) under one identifier because both are ultimately answering
"What's happening in our data?" with different emphasis.

2. TERMINOLOGY & SYNONYMS
================================================================================

The entity identified by UBADA is referred to by different terms:

DATA ANALYST
  - Definition: Technical professional analyzing claims/member data
  - Usage: Fraud detection, pattern analysis, quantitative insights
  - Example: "Senior Data Analyst - Fraud Detection"
  - System: Ad-hoc query tools, statistical analysis platforms

BUSINESS ANALYST
  - Definition: Professional analyzing program operations/outcomes
  - Usage: Program efficiency, quality metrics, strategic insights
  - Example: "Program Business Analyst - Medicaid Operations"
  - System: Reporting dashboards, KPI tracking, executive summaries

FRAUD ANALYST
  - Definition: Analyst specialized in fraud detection
  - Usage: Identifying suspicious patterns, claim aberrations
  - Example: "Medicaid Fraud Analyst"
  - System: Fraud detection algorithms, pattern matching tools

PROGRAM EVALUATOR
  - Definition: Analyst evaluating program performance
  - Usage: Assessing efficiency, quality, outcomes
  - Example: "Program Evaluator - Managed Care"
  - System: Evaluation frameworks, outcome tracking

RESEARCHER
  - Definition: Academic or policy researcher analyzing Medicaid data
  - Usage: Policy analysis, program research, publications
  - Example: "Health Services Researcher"
  - System: De-identified dataset access, research portals

AUDITOR / COMPLIANCE ANALYST
  - Definition: Analyst reviewing claims for compliance
  - Usage: Payment accuracy, policy adherence, risk assessment
  - Example: "Claims Compliance Analyst"
  - System: Claims audit tools, variance analysis

UNIFIED REFERENCE
  - All terms refer to ANALYSTS examining TORQ-e data
  - All systems map to ONE UBADA per analyst
  - UBADA defines analysis scope and data access

3. UBADA STRUCTURE
================================================================================

Format: UUID v4 + Analyst Type + Skill Level
Example: 990i2833-j73f-85h8-e150-880099884444
Analyst Type: FRAUD (fraud detection), BUSINESS (program analysis), RESEARCH
Skill Level: JUNIOR (supervised), SENIOR (independent), LEAD (oversight)

Length: 36 characters UUID + 1-10 character analyst type + skill level
Pattern: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx|TYPE|LEVEL

Generation:
  - Created when analyst role is authorized
  - Algorithm: Cryptographically secure random generation (UUID v4)
  - Analyst type and skill level assigned based on role
  - Immutable once assigned

Storage:
  - Primary database: TORQ-e Analyst Identity Database
  - Indexed by UBADA, Analyst Name, Skill Level
  - Encrypted at rest
  - Audit logging of all analyses performed

4. ANALYST TYPES & SKILL LEVELS
================================================================================

ANALYST TYPE 1: FRAUD ANALYST

JUNIOR Fraud Analyst:
  - Access: Can query claims data under supervision
  - Skills: Learning fraud detection patterns, basic SQL
  - Reports: Generate predefined fraud detection reports
  - Flags: Cannot independently flag claims for investigation
  - Escalation: Supervisor reviews and approves all findings
  - Certification: Healthcare fraud detection training required
  - Data access: De-identified claims data, basic metrics
  - Cannot: Access identifiable member/provider data alone

SENIOR Fraud Analyst:
  - Access: Full access to claims data including identifiable info
  - Skills: Advanced fraud detection, statistical modeling, SQL expertise
  - Reports: Create custom fraud detection queries and analyses
  - Flags: Can independently flag claims/providers for investigation
  - Escalation: Direct to Medicaid Inspector General
  - Certification: Healthcare fraud certification, advanced analytics
  - Data access: Full claims data, member/provider identifiers, networks
  - Can: Access all data necessary for investigation

LEAD Fraud Analyst:
  - Access: Full access plus supervisory oversight
  - Skills: Expert-level fraud detection, ML capabilities
  - Reports: Design and deploy new fraud detection models
  - Flags: Approves and escalates investigations
  - Escalation: Direct to leadership, law enforcement coordination
  - Certification: Fraud examination (CFE), healthcare expertise
  - Data access: Full access including sensitive investigation files
  - Can: Define fraud detection strategy, train team, oversee cases

---

ANALYST TYPE 2: BUSINESS ANALYST

JUNIOR Business Analyst:
  - Access: Standard dashboard access, predefined reports
  - Skills: Learning program operations, basic reporting
  - Reports: Generate predefined program metrics and reports
  - Analysis: Cannot modify report parameters independently
  - Escalation: All analyses reviewed by supervisor
  - Certification: Program knowledge training required
  - Data access: Aggregate data, summary metrics, program KPIs
  - Cannot: Access individual member/provider data

SENIOR Business Analyst:
  - Access: Custom dashboards, ad-hoc report generation
  - Skills: Program analysis, outcome evaluation, SQL for reports
  - Reports: Create custom reports, analyze program operations
  - Analysis: Can independently analyze program efficiency
  - Escalation: Present findings to program directors
  - Certification: Master's degree or equivalent; program expertise
  - Data access: Aggregate data with identifiers for program analysis
  - Can: Deep dive into program performance, identify trends

LEAD Business Analyst:
  - Access: Full access to program data, system configuration
  - Skills: Expert program evaluation, predictive analysis, visualization
  - Reports: Design KPI frameworks, create executive dashboards
  - Analysis: Provides strategic insights to leadership
  - Escalation: Direct to Deputy Commissioner, senior leadership
  - Certification: Healthcare management background, analytics expertise
  - Data access: Full access for program-level analysis
  - Can: Define program metrics, evaluate policy impacts, guide improvements

---

ANALYST TYPE 3: RESEARCH ANALYST

JUNIOR Research Analyst:
  - Access: Limited de-identified dataset for research
  - Skills: Learning research methodology, data handling
  - Research: Conduct research under faculty/PI supervision
  - Publications: Contributing author on research papers
  - Data: Only de-identified, aggregated data
  - Cannot: Access identifiable member/provider data
  - Approval: All research requires IRB and sponsor approval

SENIOR Research Analyst:
  - Access: De-identified research datasets, advanced analytics
  - Skills: Research design, statistical analysis, publication
  - Research: Lead research projects under institutional oversight
  - Publications: First/lead author on research papers
  - Data: De-identified data matching research protocol
  - Can: Access limited identifiable data if IRB-approved
  - Approval: Research protocol must be IRB-approved

LEAD Research Analyst / Principal Investigator:
  - Access: Full authority over research dataset access
  - Skills: Research leadership, policy analysis, national reputation
  - Research: Design and lead major research initiatives
  - Publications: Lead/sole author on major publications
  - Data: Access approved by IRB and program leadership
  - Can: Negotiate data access terms, publish findings
  - Approval: IRB approval plus program leadership sign-off

BUSINESS ANALYST Data Access:

9. FRAUD DETECTION & ANOMALY DETECTION
================================================================================

Fraud Detection Algorithms (Run by SENIOR+ analysts):
  1. Duplicate Billing Detection
     - Same service billed multiple times
     - Multiple providers billing same service same date
     - Alert: Flag for manual review

  2. Billing Pattern Anomalies
     - Provider billing 5x normal volume for service
     - Unusual service combinations (never billed together)
     - Billing outside hours of operation
     - Alert: Investigate provider and review claims

  3. Member Utilization Anomalies
     - Member seeing 10+ different providers same month (doctor shopping)
     - Member traveling between counties for identical services
     - Member claiming services from deceased provider
     - Alert: Verify member identity, review claims

  4. Network Analysis
     - Related providers billing same patients
     - Shell provider paying kickbacks to referring provider
     - Phantom network (claimed but non-existent provider network)
     - Alert: Network verification, investigation

  5. Financial Red Flags
     - Charges wildly above/below market rate
     - Provider receiving payment despite exclusion status
     - Payment to non-enrolled provider
     - Alert: Claims hold, investigation

Cost-Benefit Analysis:
  - Focus investigations on high-dollar potential fraud
  - Fraudulent claims > $10K per case trigger automatic escalation
  - Serial small frauds ($1K × 100 claims) also escalated
  - Low-value anomalies reviewed quarterly

10. ANALYSIS AUDIT & OUTPUT MANAGEMENT
================================================================================

All Analysis Outputs Tracked:
  - What report/analysis was run
  - By which UBADA analyst
  - What data was queried
  - How many records were accessed
  - When analysis was conducted
  - Who viewed/accessed the results
  - What findings were documented

Report Retention:
  - Fraud findings: Archived 7 years (statute of limitations)
  - Program reports: Archived 5 years
  - Research datasets: Per IRB approval (typically 5-7 years)
  - Audit trails: Permanent (never deleted)

Data Export Controls:
  - Analysts cannot export raw data to personal devices
  - Reports can only be exported in approved formats
  - All exports logged and tracked
  - Sensitive findings require supervisor approval before export

Quality Assurance:
  - 10% of fraud findings independently reviewed
  - All LEAD-level analyses verified before escalation
  - Statistical accuracy of analyses spot-checked
  - Analyst performance metrics tracked

11. TWO-FACTOR AUTHENTICATION & UBADA
================================================================================

2FA Requirements:
  - Mandatory for all UBADA analysts
  - Authenticator app (TOTP)
  - Phone number verified
  - Backup phone strongly recommended

2FA Enrollment:
  1. Analyst enrolls in authenticator app
  2. App generates time-based code
  3. Phone verified via SMS/call
  4. Backup phone can be added
  5. 2FA linked to UBADA

2FA & Data Access:
  - Lost phone = automatic lock from sensitive data
  - Re-verification required before access restored
  - High-value analysis (>$10K fraud finding) requires 2FA + approval

2FA Breach / Account Compromise:
  - Analyst reports compromised 2FA
  - Account locked immediately
  - All recent analyses reviewed for integrity
  - Re-enrollment required with verification
  - UBADA flagged for investigation
  - Access restored after clearance
